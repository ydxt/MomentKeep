# 数据存放策略 - 对齐阶段

## 1. 项目上下文分析

### 1.1 现有项目结构

- **项目名称**: MomentKeep
- **技术栈**: Flutter
- **架构模式**: 分层架构（domain, presentation, services）
- **依赖关系**:
  - 数据库: SQLite
  - 状态管理: BLoC
  - UI: Material Design 3

### 1.2 现有代码模式

- **数据库服务**: `lib/services/user_database_service.dart`
- **实体类**: `lib/domain/entities/`
- **状态管理**: `lib/presentation/blocs/`
- **UI页面**: `lib/presentation/pages/`

### 1.3 业务域理解

- **核心功能**:
  - 用户管理（买家、卖家、管理员）
  - 电商系统（商品、订单、交易）
  - 智能日记
  - 习惯打卡
  - 待办事项

## 2. 需求理解确认

### 2.1 原始需求

> 目前数据存放问题，由于目前无服务器，我们假定用户设置目录下的cloud目录为服务器目录，用于存放数据库及文件等，哪些数据必须放在服务器，哪些数据需要放在本地？可以参考线上商店、日记系统等待

### 2.2 边界确认

- **任务范围**: 制定数据存放策略，明确服务器（cloud目录）和本地存储的数据类型
- **技术约束**: 无实际服务器，使用本地目录模拟服务器
- **业务约束**: 需同时支持电商系统和个人管理系统的存储需求

### 2.3 需求理解

- **服务器目录**: `用户设置目录/cloud/` - 模拟服务器存储
- **本地目录**: `用户设置目录/` - 本地存储
- **数据分类**: 需要明确区分哪些数据必须放在服务器，哪些数据适合放在本地
- **参考系统**: 线上商店（电商数据集中管理）、日记系统（个人数据本地存储）

### 2.4 疑问澄清

- **数据同步机制**: 如何实现本地与"服务器"（cloud目录）之间的数据同步？
- **数据安全**: 如何确保敏感数据的安全性？
- **存储限制**: 是否有存储容量限制需要考虑？
- **离线使用**: 如何确保在无网络情况下的正常使用？

## 3. 智能决策策略

### 3.1 决策依据

- **数据特性**: 敏感程度、共享需求、实时性要求
- **业务需求**: 电商系统需要集中管理，个人系统需要隐私保护
- **技术可行性**: 本地存储性能优势，服务器存储集中管理优势

### 3.2 初步决策

- **服务器存储**: 用户认证数据、电商交易数据、系统配置
- **本地存储**: 个人隐私数据、缓存数据、离线数据
- **同步策略**: 实时同步关键数据，定期同步非关键数据，手动同步大文件

## 4. 关键决策点

1. **数据分类标准**: 如何定义哪些数据必须放在服务器，哪些数据放在本地？
2. **同步机制设计**: 如何实现本地与cloud目录之间的数据同步？
3. **数据安全措施**: 如何保护敏感数据？
4. **存储优化策略**: 如何优化存储资源使用？
